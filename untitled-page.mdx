title: "Responses"
---

The ArcWeather API returns data in multiple formats depending on the endpoint and request configuration. While most endpoints default to JSON, some support alternative formats such as XML or CSV, and others may return binary assets like images.

## Response Formats

### Default: JSON

All endpoints return **JSON by default** unless another format is explicitly requested.

```text
{
  "status": "ok",
  "data": { }
}
```

### Optional Formats

Some endpoints support additional formats:

- **XML** — Typically for legacy or integration-based use cases
- **CSV** — Useful for data exports and analytics workflows

To request a different format, refer to the specific endpoint documentation. Format selection may be controlled via query parameters or headers depending on the endpoint.

> Not all endpoints support all formats. Always verify supported response types per endpoint.

### Binary Responses (Images & Tiles)

Endpoints related to weather visualization (such as radar or lightning tiles) return binary image formats:

- **PNG** — Default for most tile-based endpoints
- **WebP** — Used for optimized performance where supported

Example:

```text
GET /v1/radar/tiles/{z}/{x}/{y}.png
```

These responses will return raw image data, not JSON.

## Rate Limit Headers

Every API response includes rate limiting information in the headers. These headers help you track usage and avoid exceeding your allocated limits.

### Common Headers

```text
X-RateLimit-Limit: 1000
X-RateLimit-Remaining: 742
X-RateLimit-Reset: 1712083200
```

- **X-RateLimit-Limit** — Total number of requests allowed in the current window
- **X-RateLimit-Remaining** — Number of requests remaining
- **X-RateLimit-Reset** — UNIX timestamp when the limit resets

### Best Practices

- Monitor remaining requests to avoid throttling
- Implement retry logic when approaching limits
- Cache responses when possible to reduce usage

## Errors

ArcWeather uses standard HTTP status codes to indicate request success or failure. When an error occurs, the API will return a structured response with details.

### Error Response Format

```text
{
  "status": "error",
  "error": {
    "code": 401,
    "message": "Unauthorized"
  }
}
```

## Common Error Codes

### 400 — Bad Request

The request was malformed or missing required parameters.\
**What to do:**

- Validate request structure
- Check required fields and query parameters

### 401 — Unauthorized

Authentication failed due to missing or invalid credentials.\
**What to do:**

- Ensure both `Authorization` and `X-App` headers are included
- Verify API key and app identifier are correct

### 403 — Forbidden

The request is valid but not allowed.\
**What to do:**

- Check API key permissions
- Confirm the endpoint is enabled for your account

### 404 — Not Found

The requested resource or endpoint does not exist.\
**What to do:**

- Verify the endpoint path
- Check parameters such as IDs or coordinates

### 429 — Too Many Requests

Rate limit has been exceeded.\
**What to do:**

- Wait until `X-RateLimit-Reset`
- Reduce request frequency
- Implement backoff and retry logic

### 500 — Internal Server Error

An unexpected error occurred on the server.\
**What to do:**

- Retry the request
- Contact support if the issue persists

### 503 — Service Unavailable

The service is temporarily unavailable.\
**What to do:**

- Retry after a short delay
- Monitor system status if applicable

## Summary

- JSON is the default response format
- Some endpoints support XML or CSV
- Tile and visualization endpoints return PNG or WebP images
- All responses include rate limit headers
- Errors follow standard HTTP status codes with structured messages

Always refer to each endpoint’s documentation for exact response formats and supported options.
